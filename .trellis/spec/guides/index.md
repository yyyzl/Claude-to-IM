# 思考指南

> **目的**：在动手前把容易遗漏的问题想清楚，减少跨层 bug、重复实现和后续返工。

---

## 为什么需要思考指南

多数 bug 和技术债不是因为不会写，而是因为前面没想全：

- 没想清边界在哪里，结果跨层契约断裂
- 没想到已有实现，结果复制粘贴越来越多
- 没想到极端输入，结果运行时才炸
- 没想到后续维护者，结果代码越来越难懂

这些指南的作用，就是在编码前逼自己先问对问题。

## 快速触发条件

### 何时优先读跨层思考指南

- [ ] 改动会穿过 3 层以上
- [ ] 共享类型、配置 key、消息格式会变化
- [ ] 多个模块都要同步改
- [ ] 你不确定某段逻辑究竟该落在哪一层
- [ ] 你把用户生成的内容（diff、源码、配置）注入到模板或命令中
- [ ] 你的组件输出会被下游外部系统消费（注意大小/格式限制）
- [ ] 你的函数参数来自不同的过滤/聚合管道（注意基数是否一致）
- [ ] 你在状态机的汇合点写入固定值（注意所有到达路径）

→ 读 [跨层思考指南](./cross-layer-thinking-guide.md)

### 何时优先读代码复用思考指南

- [ ] 你准备新建 helper 或工具函数
- [ ] 你看到已有类似命令、渲染、校验或配置逻辑
- [ ] 你要改常量、平台限制或配置项
- [ ] 你准备复制一段看起来“差不多”的实现

→ 读 [代码复用思考指南](./code-reuse-thinking-guide.md)

### 何时优先读设计模式指南

- [ ] 你要扩平台适配器、bridge manager 或 workflow engine
- [ ] 你不确定新能力应该挂在 context、adapter、delivery 还是 workflow profile
- [ ] 你看到 `globalThis`、注册表、profile、checkpoint 这类结构，但不确定它们是不是稳定约定
- [ ] 你需要理解或扩展某个架构模式，而不是单点修复一段逻辑

→ 读 [设计模式指南](./design-patterns-guide.md)

## 修改前铁律

> **改任何值、类型、命令或配置前，先搜索。**

```bash
rg "value_to_change" .
```

这是当前仓库里最划算的习惯之一，可以显著减少“漏改某一处”的问题。

---

## 如何使用本目录

1. 编码前先读与你任务最相关的一份指南
2. 编码中如果发现跨层、复用或架构骨架问题，再回头补读另一份
3. 遇到新坑后，把结论补回对应指南，沉淀成以后能复用的约束

---

## 当前仓库的特别提醒

这个仓库最常见的跨层和复用问题集中在：

- `src/lib/bridge/` 与 `scripts/` 的边界
- `host.ts`、`types.ts` 的共享契约
- 命令解析、帮助文案、测试和配置 key 的同步更新
- 多平台适配器之间的重复实现

---

**核心原则**：前面多想 30 分钟，后面少调 3 小时。
