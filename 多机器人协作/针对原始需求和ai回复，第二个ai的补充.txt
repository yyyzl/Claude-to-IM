 我的深度分析与意见

  一、对那个 AI 方案的评价：方向对了，但有四个盲点

  总评：70分。核心判断（做 Skill，用 codeagent-wrapper 调 Codex）是务实的，但分析不够深。

  盲点1：忽略了飞书群聊场景

  你的原始问题是"一个群里同时有两个机器人"，但 AI 只给了终端 Skill 方案。群聊场景才是你日常使用频率最高的入口，需要考虑。

  盲点2：低估了 Prompt 构造的复杂度

  AI 说"核心做两件事"很轻巧，但你三种工作流的 prompt 构造逻辑完全不同：

  ┌───────────┬──────────────┬────────────────────────────────────────────────────┐
  │  工作流   │  信息流形态  │                  Prompt 构造策略                   │
  ├───────────┼──────────────┼────────────────────────────────────────────────────┤
  │ Spec 审查 │ 螺旋累积     │ 每轮叠加：原始spec + 所有历史意见 + 所有Claude反馈 │
  ├───────────┼──────────────┼────────────────────────────────────────────────────┤
  │ 开发      │ 流水线推进   │ 每轮替换：当前进度摘要 + 新任务指令                │
  ├───────────┼──────────────┼────────────────────────────────────────────────────┤
  │ 代码审查  │ 独立交叉验证 │ 每轮独立：只有 spec/plan + 当前代码，无历史        │
  └───────────┴──────────────┴────────────────────────────────────────────────────┘

  这三种策略需要一个模板系统，不是写死在代码里的。

  盲点3：对 SO 的评估过于绝对

  AI 说"改造需要重写 60%，不值得"。我完整看了 SO 的代码后认为：实际需要改动的是 Runner 路由 + 工作流状态机，约 35-40%。SO 的 RunPack 工件体系、事件审计系统、Web UI
  都是可以直接复用的基础设施。不应该完全放弃。

  盲点4：遗漏了工件持久化

  如果只做 Skill（在 Claude Code 上下文中运行），历史信息会随着上下文压缩而丢失。必须有磁盘持久化。这正是 RunPack 思路的价值。

  ---
  二、你的工作流设计直觉非常好——背后是严肃的方法论

  你可能没意识到，但你的三种工作流分别对应了经典的决策科学方法：

  工作流1（Spec审查）= 德尔菲法（Delphi Method）
  - Codex 清空上下文 = 匿名独立评估（避免锚定效应）
  - 每轮累积历史 = 结构化反馈
  - Claude 酌情采纳 = 协调者角色
  - 直到收敛 = 德尔菲法的终止条件

  工作流2（开发）= Manager-Worker 模式
  - Claude 保持上下文 = Manager 的全局视野
  - Codex 清空+进度 = Worker 只需知道当前任务
  - 流水线推进 = 经典的任务分派

  工作流3（代码审查）= 对抗性验证（Adversarial Review）
  - 双方都清空 = 避免确认偏差，这是你最精妙的设计
  - 独立判断 = 从 spec/plan 出发，不受开发叙事影响
  - 三步循环（审查→判断→修复）= 完整的反馈闭环

  这三种模式的底层统一抽象是：结构化对抗性协作（Structured Adversarial Collaboration）——两个模型在设计好的信息隔离和信息共享策略下协作，通过循环趋向收敛。

  三、我认为最合理的路径（三阶段）

  第一阶段：文件驱动的半自动化（最快见效，1-2天）

  不改任何代码。只做一件事：标准化工件目录 + Prompt 模板。

  .workflow/
    templates/
      spec-review/
        codex-input.md.hbs      # "请审查以下 spec... 前几轮意见：{{history}}"
        claude-decision.md.hbs  # "Codex 意见如下... 请决策"
      dev-loop/
        codex-task.md.hbs       # "进度：{{progress}} 下一步：{{task}}"
      code-review/
        codex-audit.md.hbs      # "背景：{{spec}} 请审查 {{diff}}"
    runs/
      spec-review-20260319/
        meta.json               # { workflow: "spec-review", round: 2, status: "in-progress" }
        round-1-codex-input.md
        round-1-codex-output.md
        round-1-claude-decision.md
        round-2-codex-input.md  # 由模板自动生成，引用前几轮文件
        ...

  即使手动操作，用模板渲染 prompt 也比每次从头组织快 5 倍以上。这就是你当前最痛的那个"手动粘贴历史意见"的直接解。

  第二阶段：Claude Code Skill 全自动化（3-5天）

  做三个 Skill（/spec-review, /dev-loop, /code-review），核心循环：

  读取模板 + 累积历史文件
      → 渲染 codex-input.md
      → codeagent-wrapper 调 Codex
      → 捕获输出到 codex-output.md
      → 展示给 Claude/用户决策
      → 保存 claude-decision.md
      → 检查终止条件
      → 未终止？回到第一步

  关键设计点：
  - 每轮产物落盘，不依赖上下文窗口保持历史
  - 决策环节暂停，等用户确认（半自动，不是全自动）
  - 终止条件可配（LGTM/最大轮次/无新问题）

  第三阶段：桥接层集成（后续）

  在 Claude-to-IM 中增加 /workflow 命令族，群聊内触发：

  用户 @Claude: /workflow spec-review spec.md plan.md
  Claude: 🔄 启动 Spec 审查循环（最多5轮）
  Claude: [第1轮] Codex 审查意见：1. xxx 2. xxx
  Claude: [第1轮] 我的决策：采纳1，拒绝2。已更新 spec.
  Claude: [第2轮] Codex 审查意见：无新问题
  Claude: ✅ 审查通过，共2轮

  在群聊场景中，Codex 不需要作为独立机器人发言，Claude 桥接在后台通过 codeagent-wrapper 调用即可。一个机器人对外，后台编排。

  四、关于 Session Orchestrator 的联动建议

  不建议直接改造 SO，但建议从 SO 借鉴三样东西：

  1. RunPack 目录结构 —— 直接用于 .workflow/runs/ 的设计
  2. 事件流 ndjson 格式 —— 记录每轮的输入/输出/决策事件，便于审计和回溯
  3. 熔断机制 —— 同一问题连续 N 轮未收敛 → 自动暂停，提醒用户介入

  SO 作为"参考实现"保留，未来如果需要产品化（比如给团队用、或者支持 3+ 模型协作），再考虑把 SO 扩展为通用引擎。

  五、对"飞书群里两个机器人怎么用"的直接回答

  短期最务实的方式：
  - Claude 机器人 = 主力，日常对话、需求探讨、代码编写都用它
  - Codex 机器人 = 独立审查者，只在需要"第二意见"时使用
  - 人工在两者之间中继（现状），但用模板化的 prompt 降低中继成本

  中期理想方式：
  - Claude 机器人作为唯一入口
  - 用户发 /workflow spec-review 这样的命令
  - Claude 后台自动调 Codex，群聊里只看到 Claude 的消息（但内容包含 Codex 的意见）
  - 两个机器人在技术上仍然独立运行，但编排逻辑统一在 Claude 侧

  为什么不让两个机器人在群聊里"对话"？
  因为飞书的机器人 API 限制——机器人通常只响应用户消息，不响应其他机器人的消息。让两个机器人自动对话需要额外的消息路由层，复杂度高且收益有限。不如 Claude 在后台直接调用 Codex 的
  CLI。

  ---
  总结一句话

  那个 AI 的方向是对的（Skill + codeagent-wrapper），但关键缺失是模板系统和工件持久化。你最该优先做的不是代码编排引擎，而是标准化你的 Prompt 模板和工件目录——这才是消除 80%
  重复劳动的杠杆点。SO 不用改造但值得借鉴其 RunPack 思路。
