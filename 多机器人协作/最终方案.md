# 多机器人协作 — 最终方案（v2 修正版）

> 综合 5 个 AI 方案 + 2 份实施计划 + 2 份独立评审意见，提炼共识、裁决分歧后形成的统一方案。
>
> v1: 2026-03-19 · v2: 2026-03-19（合入两份评审意见的 12 项修正）

---

## 一、5 个 AI 的共识点（高置信度，全部采纳）

1. **Claude = 控制面，Codex = 执行/审查面** — 不要对等协作
2. **一个 Bot 做门面** — 群里只有一个 Bot 发言，后端双引擎编排
3. **结构化 Pack 交接** — 不靠聊天记录搬运信息，靠标准化工件包
4. **工件必须落盘** — 不依赖上下文窗口保持历史，磁盘是事实源
5. **动态终止** — "有增量信息才继续"而不是"固定 N 轮"
6. **分阶段实施** — 先手动验证协议，再自动化，再接入平台

## 二、核心架构

```
┌──────────────────────────────────────┐
│       飞书 / Telegram / Discord 群     │
│  （唯一的人机界面，一个 Bot 门面）       │
└──────────────┬───────────────────────┘
               │ 消息 / 按钮回调
┌──────────────▼───────────────────────┐
│     Claude-to-IM Bridge（交互层）      │
│  • /workflow 命令路由                  │
│  • 进度推送 + 卡片渲染                 │
│  • 飞书按钮 → approve / reject        │
│  ⚠ Bridge 核心层不放编排逻辑           │
└──────────────┬───────────────────────┘
               │ 内部调用
┌──────────────▼───────────────────────┐
│     Workflow Engine（编排层）           │  ← 独立模块，不在 bridge 核心层内
│  • WorkflowDefinition（循环步骤链）     │
│  • PackBuilder（从 Store 组装 Pack）   │
│  • PromptAssembler（模板渲染）          │
│  • ModelInvoker（Claude SDK / Codex CLI）│
│  • TerminationJudge（动态终止判断）     │
│  • IssueLedger（决策台账）             │
└───────┬─────────────┬────────────────┘
        │             │
   Claude API    Codex CLI
  (kept/fresh)  (天然 fresh)
        │             │
┌───────▼─────────────▼────────────────┐
│     Artifact Store（持久层）           │
│  .claude-workflows/{run-id}/          │
│  ├── meta.json                        │
│  ├── spec.md / plan.md                │
│  ├── issue-ledger.json                │
│  ├── events.ndjson                    │
│  └── rounds/                          │
│      ├── R1-codex-review.md           │
│      ├── R1-claude-decision.md        │
│      └── ...                          │
└───────────────────────────────────────┘
```

### 架构边界（v2 新增）

| 层 | 职责 | 不该做的事 |
|---|---|---|
| **Bridge 交互层** | 命令路由、消息推送、按钮回调 | ❌ 不放工作流状态机、不做 Pack 组装 |
| **Workflow Engine** | 编排、终止判断、Pack 构建、模型调用 | ❌ 不直接操作 IM API |
| **Artifact Store** | 读写文件、事件日志 | ❌ 不做业务判断 |

> **为什么 Workflow Engine 不进 Bridge 核心层？** Bridge 的定位是 host-agnostic 的消息桥接（见 `src/lib/bridge/README.md`），把业务编排塞进去会破坏它的通用性。Workflow Engine 应作为 `src/lib/workflow/` 独立模块，Bridge 只负责命令入口和消息推送。

### 设计原则

- **群消息是给人看的**，不是给机器看的。机器之间的协作在后端通过 Pack / API 完成。
- **Claude 永远是控制面**，持有全局状态、理解意图、做裁决和收口。
- **Codex 永远是无状态工人/挑战者**，按 Pack 执行，新窗口 = 天然清空上下文。
- **Issue Ledger 是单一事实源**，避免"Claude 记得 A，Codex 盯着 B，你脑子里还有 C"。

## 三、三个核心工作流

### 工作流 1：Spec / Plan 审查（德尔菲法）

**角色分工**：Claude 保持上下文做裁决，Codex 清空上下文做盲审。

```
Claude(kept) ←→ Codex(fresh_with_pack)

每轮 Codex 收到的 SpecReviewPack:
{
  spec,
  plan,
  issue_ledger,            // 已裁决的完整台账
  unresolved_issues,       // 只传未解决的
  context_files,           // 需要阅读但不修改的参考文件
  round_summary            // "上轮新增 3 个问题，接受 2 个，拒绝 1 个"
}

每轮 Codex 输出:
  结构化 findings: [{issue, severity, evidence}]

Claude 裁决:
  逐条 accept / reject / defer，写入 Issue Ledger，更新 spec/plan
```

**终止条件（动态判断）**：

| 条件 | 触发 |
|------|------|
| Codex 输出 LGTM | 立即终止 |
| 连续 2 轮无新增高严重度问题 | 终止 |
| 只剩风格性建议 | 终止 |
| 同一 issue 连续 2 轮被提出又被拒绝 | 升级为人工介入 |
| 达到最大轮次 | 常规 3 轮 / 架构级大改 5 轮 |

**Claude 上下文压缩策略（v2 新增）**：

当 token 数超过 60% 窗口容量、或审查进入第 4 轮时，Claude 将当前状态压缩为：
1. 最新版 spec/plan 全文
2. Issue Ledger 摘要（仅 open + accepted 条目）
3. 最近一轮的完整交互内容
4. 丢弃中间轮次的原始文本

**关键设计**：不传完整聊天历史，只传 Issue Ledger + 未决问题 → 解决 prompt 膨胀问题。

---

### 工作流 2：开发（Manager-Worker）

**角色分工**：Claude 保持上下文做项目经理，Codex 清空上下文做执行者。

```
Claude(kept) → Codex(fresh_with_pack)
```

**TaskPack（Claude → Codex）**：

```json
{
  "goal": "当前 WorkItem 目标",
  "scope": "允许修改的文件范围",
  "context_files": "需要阅读但不修改的参考文件列表",
  "acceptance": "验收标准",
  "forbidden": "禁止修改的范围",
  "completed_summary": "已完成的工作摘要",
  "dependencies": "依赖的上游产物",
  "workspace_strategy": "branch / worktree 隔离策略"
}
```

**DeliveryPack（Codex → Claude）**：

```json
{
  "changes": "改了什么",
  "files_modified": ["..."],
  "test_results": "测试结果",
  "risks": "遗留风险",
  "next_suggestion": "建议下一步"
}
```

**并行隔离策略（v2 新增）**：

独立的 WorkItem 可以并行多个 Codex，但必须定义隔离方式：

| 策略 | 适用场景 | 实现方式 |
|------|---------|---------|
| **分支隔离** | 多数场景 | 每个 WorkItem 在独立 feature branch 上执行 |
| **Worktree 隔离** | 需同时编译/测试 | `git worktree add` 创建独立工作目录 |
| **文件所有权** | 轻量并行 | TaskPack.scope 严格互斥，不允许重叠 |

规则：
- 默认使用分支隔离（最安全）
- scope 有重叠的 WorkItem **禁止并行**，必须串行
- Claude 负责在拆 WorkItem 时标注隔离策略

**关键设计**：

- 粒度是 **WorkItem**，不是"每轮做一小部分"
- Claude 只 **拆工 + 验收**，不微观遥控
- WorkItem 切分质量决定协作效率

**终止条件**：所有 WorkItem 完成。

---

### 工作流 3：代码审查（对抗性验证）

**角色分工**：Claude 和 Codex **都清空上下文**，避免确认偏差。

**每一步的上下文策略（v2 显式标注）**：

```
Step 1: Codex 盲审 (fresh_with_pack)
  输入: ReviewPack{spec, plan, code_snapshot, context_files}
  输出: issue list
  上下文: 全新窗口，只有 Pack 内容

Step 2: Claude 裁决 (fresh_with_pack)
  输入: Codex 的 issue list + spec/plan + code_snapshot
  输出: 逐条严重度判断，过滤误报，生成修复指令
  上下文: 全新窗口，防止开发阶段的确认偏差

Step 3: Codex 修复 (fresh_with_pack) — 与 Step 1 不同的新窗口
  输入: 裁决后的 accepted issues + 修复指令 + 相关代码
  输出: 代码修改
  上下文: 全新窗口，只看被接受的问题

复核循环:
Step 4: Codex 快审 (fresh_with_pack) — 又一个新窗口
  输入: diff + spec/plan（不用全量审查）
  输出: 新发现的问题（如有）

Step 5: 有新问题 → 回到 Step 2；无新问题 → 结束
```

> 注意：Step 1、3、4 的 Codex 是三个不同的新窗口。每个 Step 都是 fresh_with_pack，确保无偏见。

**关键设计**：

- **分离"发现问题"和"决定要不要改"** — 避免两个模型互相放大噪声
- Claude 裁决时 **过滤误报** — 不是所有 Codex 意见都要执行
- 修复后用 **diff 级快审** — 不用每次全量审查
- 最大 **3 轮循环**

**终止条件**：

| 条件 | 触发 |
|------|------|
| 复核无新增问题 | 终止 |
| 只剩低风险问题 | 终止 |
| 同一 issue 连续 2 轮反复提出又被拒绝 | 升级为人工介入 |
| 达到最大 3 轮循环 | 强制终止 |

## 四、核心数据结构

### Issue Ledger（决策台账）

所有工作流共用的单一事实源。

```json
{
  "run_id": "spec-review-20260319",
  "issues": [
    {
      "id": "ISS-001",
      "round": 1,
      "raised_by": "codex",
      "severity": "high",
      "description": "缺少错误处理策略",
      "evidence": "spec.md L45-L60",
      "status": "accepted",
      "decided_by": "claude",
      "decision_reason": "确实遗漏，已补充",
      "resolved_in_round": 2,
      "repeat_count": 0
    },
    {
      "id": "ISS-002",
      "round": 1,
      "raised_by": "codex",
      "severity": "low",
      "description": "建议用策略模式",
      "status": "deferred",
      "decided_by": "claude",
      "decision_reason": "当前规模不需要，后续优化",
      "repeat_count": 0
    }
  ]
}
```

**v2 新增字段**：`repeat_count` — 同一 issue 被反复提出的次数。当 `repeat_count >= 2` 且 status 仍为 rejected，自动触发人工介入。

**价值**：

- 避免多方信息分叉
- 每轮 Codex 只需看未解决 issues，不需重读完整历史
- 审计可追溯 — 每个决策都有理由
- 自然支持终止判断 — "还有未解决的高严重度问题吗？"
- **死循环检测** — repeat_count 防止两个模型在同一问题上无限争执

### Pack 通用结构（v2 新增 context_files）

所有 Pack 类型都包含 `context_files` 字段：

```json
{
  "context_files": [
    "src/lib/bridge/types.ts",
    "src/lib/bridge/host.ts"
  ]
}
```

> `scope` = 允许修改的文件，`forbidden` = 禁止修改的文件，`context_files` = 需要阅读但不修改的参考文件。三个字段语义互斥。

### Artifact Store 目录结构

```
.claude-workflows/
└── {run-id}/
    ├── meta.json                  # 工作流元信息 + 当前状态（断点续传）
    ├── spec.md                    # 原始 spec（版本化）
    ├── spec-v2.md                 # 修改后的 spec
    ├── plan.md                    # 原始 plan
    ├── plan-v2.md                 # 修改后的 plan
    ├── issue-ledger.json          # 决策台账
    ├── events.ndjson              # 事件流日志（借鉴 SO）
    └── rounds/
        ├── R1-codex-review.md     # 第 1 轮 Codex 审查原文
        ├── R1-claude-decision.md  # 第 1 轮 Claude 裁决
        ├── R2-codex-review.md
        └── ...
```

## 五、Session Orchestrator 定位

### 最终判断：做后台底座，不做前台引擎

综合 AI-3/4/5 的分析（他们都实际看了 SO 代码），SO 现状：

| 已具备（可复用） | 缺失（需增量扩展，~400 行） |
|---|---|
| RunPack 工件存储 | Actor 概念（claude_control / codex_review / codex_exec） |
| human_review 闸门 | 循环步骤链（goto + condition） |
| work_items 管理 | 上下文策略标注（keep / fresh / fresh_with_pack） |
| window / handoff 语义 | Pack 文件标准 |
| 事件流审计 | Issue Ledger |

### 联动策略

```
阶段性策略：
  现在 → Workflow Engine 在 Claude-to-IM 内作为独立模块实现（src/lib/workflow/），SO 暂不改动
  验证 → 协议和模板在实际使用中迭代稳定
  成熟 → 把循环编排能力下沉到 SO，SO 做状态仓库 + 审计仓库

从 SO 借鉴三样东西：
  1. RunPack 目录结构 → .claude-workflows/ 设计
  2. 事件流 ndjson 格式 → events.ndjson 审计日志
  3. 熔断机制 → 同一问题连续 N 轮未收敛 → 自动暂停
```

### SO 兼容性契约（v2 新增）

为避免未来下沉到 SO 时重写两遍，现在就要提前对齐以下格式：

| 格式 | 现在定义 | SO 下沉时映射 |
|------|---------|-------------|
| Pack JSON | TaskPack / ReviewPack / SpecReviewPack | → SO RunPack 的 observation_pack 扩展 |
| Issue Ledger | issue-ledger.json | → SO work_items.csv 的结构化升级 |
| 事件流 | events.ndjson | → SO events.ndjson（直接兼容） |
| 元信息 | meta.json | → SO snapshot.json 的超集 |

### SO "成熟"判断标准（v2 新增）

当工作流模板和 Pack 结构满足以下条件时，启动 SO 下沉：
- 在 **3+ 个真实项目**中跑通
- **连续 2 周**无结构性变更（字段增删、流程步骤变化）
- 核心模块的单元测试覆盖率 > 80%

## 六、IM 集成设计

### 命令体系

```
/workflow spec-review <spec> <plan>   → 启动 Spec/Plan 审查
/workflow dev <spec> <plan>           → 启动编排式开发
/workflow code-review <spec> <plan>   → 启动代码审查
/workflow status                      → 查看当前进度
/workflow pause                       → 暂停
/workflow resume                      → 继续
/workflow skip                        → 跳过当前步骤
```

### 消息推送策略

| 事件 | 推送到群的消息 |
|------|-------------|
| 循环开始 | "🔄 第 N 轮审查开始" |
| Codex 审查完成 | "🔍 Codex 意见: {摘要}（{N} 个问题）" |
| Claude 裁决完成 | "💡 Claude 裁决: 接受 {N} 项，拒绝 {M} 项，延后 {K} 项" |
| 需要人工介入 | "⏸️ 需要你的判断 [继续] [修改] [终止]" |
| 死循环检测触发 | "⚠️ issue ISS-XXX 连续 2 轮争议未决，需人工裁决 [接受] [拒绝] [跳过]" |
| LGTM 达成 | "✅ 审查通过！共 {N} 轮，解决 {M} 个问题" |
| 工作流结束 | "📋 完成，查看报告: {path}" |

### 核心原则

- 群消息是 **通知面板**，不是编排总线
- 只推送 **关键节点**，不推送中间过程噪音
- 支持飞书 **inline 按钮** 做 approve / reject / skip

## 八、风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 累积 prompt 超过 Codex 上下文窗口 | 只传 Issue Ledger + 未决问题，不传完整历史 |
| 审查循环不收敛（互相找茬） | 动态终止 + 硬性上限 + **死循环检测**（repeat_count ≥ 2 → 人工介入） |
| codeagent-wrapper 调用超时 | 可配超时（默认 90s）；超时跳过并记录 |
| Claude kept 模式上下文爆炸 | **ContextCompressor**：token > 60% 窗口时压缩，只保留最新 spec + Ledger 摘要 + 最近一轮 |
| 并行 Codex 互相踩文件 | **workspace_strategy**：分支/worktree/文件所有权隔离，scope 重叠禁止并行 |
| 工作流中途崩溃 | meta.json 断点续传；每步完成后自动保存 |
| IM 消息过多打扰用户 | 智能聚合事件；只推送关键节点 |
| SO 下沉时格式不兼容 | **兼容性契约**：Pack/Ledger/events 格式提前与 SO 对齐 |

## 九、不该做的事（反模式清单）

1. **不要让两个机器人做成对等群聊成员** — 看起来热闹，弱化责任边界
2. **不要把聊天记录本身当状态** — Issue Ledger 才是事实源
3. **不要把多轮当目标** — 增量信息才是价值，无增量就停
4. **不要一上来重写多智能体平台** — 先固化协议，再自动化，最后平台化
5. **不要让 Claude 微观遥控 Codex** — Claude 拆工 + 验收，Codex 在边界内自主实现
6. **不要简单改动也双审** — 只在高杠杆节点使用双模型（spec 定稿前、高风险实现、合并前审查）
7. **不要把编排逻辑塞进 Bridge 核心层** — Bridge 是通用消息桥，Workflow 独立成模块

---

## 十、一句话总结

> Claude 持有全局状态和裁决权，Codex 在短命窗口里按 Pack 执行/审查，双方围绕 Issue Ledger 单一事实源协作；飞书群只是通知面板，不是编排总线；Workflow Engine 独立于 Bridge 核心层；先固化协议和模板，再自动化，最后才考虑平台化。
